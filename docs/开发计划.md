# Cloudflare Workers AI MoA 服务开发计划

> 版本：v0.1  
> 更新时间：2026-07-16  
> 状态：开发计划初稿  
> 关联需求：[`需求文档.md`](./需求文档.md)

## 1. 开发目标

实现一个部署在 Cloudflare Worker 上、专门面向 Claude Code 的 MCP Streamable HTTP 服务。

Claude Code 通过配置 MCP Server 连接 Worker，Worker 内部通过 Cloudflare Workers AI 调用多个模型，按照 Mixture-of-Agents（MoA）架构完成多层推理，并将以下内容返回给 Claude Code：

1. 最终聚合答案；
2. 每层 proposer/aggregator 的中间 Agent 结果；
3. 模型、角色、状态、耗时和降级信息；
4. 可选的执行统计信息。

本计划以 MVP 为目标，优先完成稳定的非流式两层 MoA 链路，再进行性能、成本和质量优化。

## 2. 已确认的产品约束

| 项目 | 决策 |
|---|---|
| 唯一客户端 | Claude Code |
| 用户配置入口 | 只允许在 Claude Code 中配置 MCP Server |
| 服务协议 | MCP Streamable HTTP |
| Worker 接口 | `https://<worker-domain>/mcp` |
| 推理平台 | Cloudflare Workers AI |
| 账户计划 | 可配置，默认免费账户 |
| 预算 | 可配置，由服务端控制 |
| 推理区域 | 可配置，默认美国（US） |
| 模型 | 可配置，由服务端维护白名单 |
| 中间结果 | 必须返回，默认启用 |
| 语言 | 支持中文、英文和中英混合任务，默认自动识别 |
| 外部 MCP 工具 | 当前版本不实现 |
| 外部副作用 | 当前版本不执行 shell、文件、数据库或第三方写操作 |

## 3. 技术边界

### 3.1 Claude Code 侧

Claude Code 只负责：

- 配置 MCP Server URL；
- 根据用户任务决定是否调用 `moa_reason`；
- 发送 MCP 工具调用参数；
- 接收并展示 Worker 返回的最终答案和中间结果。

Claude Code 不负责：

- 选择具体 Workers AI 模型；
- 控制 MoA 层数和 proposer 数量的服务端硬限制；
- 直接调用 Cloudflare Workers AI；
- 执行 Worker 返回的模型输出。

### 3.2 Worker 侧

Worker 负责：

- MCP 协议处理；
- Claude Code 请求鉴权；
- 工具参数校验；
- MoA 层级编排；
- Workers AI 调用；
- 并发、超时、重试和预算控制；
- 中间结果组织、截断和脱敏；
- 错误映射和可观测性。

### 3.3 当前明确不实现

- claude.ai、Claude Desktop 或其他 MCP 客户端的兼容性保证；
- MCP 外部工具接入；
- Claude Code 本地文件读取；
- 模型输出中的隐藏思维链获取；
- 任何模型输出驱动的 shell 或任意代码执行；
- 独立 Web 管理后台；
- 在线模型训练或微调。

## 4. 目标架构

```text
Claude Code
    │
    │ MCP Streamable HTTP
    │ POST /mcp
    ▼
Cloudflare Worker
    ├── MCP Transport
    ├── Authentication
    ├── Request Validator
    ├── Budget / Rate Limit Guard
    ├── MoA Orchestrator
    │     ├── Layer 1: proposer A ─┐
    │     ├── Layer 1: proposer B ─┼─ parallel env.AI.run()
    │     ├── Layer 1: proposer C ─┘
    │     └── Layer 2: aggregator ─── env.AI.run()
    ├── Result Normalizer
    ├── Intermediate Result Builder
    └── Sanitized Trace Collector
           │
           ▼
    MCP tool result
           │
           ▼
Claude Code
```

## 6. 分阶段开发计划

## 阶段 0：项目初始化与技术验证

### 目标

建立可运行的 TypeScript Cloudflare Worker，确认 MCP 实现可以运行在 Cloudflare Workers runtime。

### 工作项

- 初始化 Worker 项目、TypeScript、Wrangler、测试和格式化工具；
- 配置 Workers AI binding；
- 验证 `env.AI` 类型和本地开发访问方式；
- 调研并选择 Cloudflare Workers 兼容的 MCP Streamable HTTP 实现；
- 确认 MCP SDK 不依赖 Node.js 专有 API、TCP socket、本地文件系统或长驻进程；
- 建立最小 `fetch` 入口和健康检查响应；
- 建立开发、测试和部署命令。

### 交付物

- 可由 `npx wrangler dev` 启动的 Worker；
- `wrangler.jsonc`；
- `src/index.ts`；
- `package.json` 和 TypeScript 配置；
- MCP SDK 选型和兼容性记录。

### 验收标准

- Worker 可以本地启动；
- 健康检查返回 200；
- Workers AI binding 配置通过类型检查；
- MCP SDK 最小示例可以在 Worker 环境执行，或已明确记录替代实现方案。

## 阶段 1：Claude Code MCP 接入

### 目标

在真实 Claude Code 环境中完成连接、初始化、工具发现和 mock 工具调用。

### 工作项

- 设计协议错误和业务错误的区分方式；
- 编写 Claude Code 配置示例。

```bash

  --header "Authorization: Bearer <token>"
```

项目级 `.mcp.json` 示例：

```json
{
  "mcpServers": {
    "moa": {
      "type": "http",
      "url": "https://<worker-domain>/mcp",
      "headers": {
        "Authorization": "Bearer ${MOA_AUTH_TOKEN}"
      }
    }
  }
}
```

### 交付物

- 可被 Claude Code 发现的 `moa_reason`；
- Bearer Token 鉴权中间件；
- `.mcp.json` 示例；
- MCP 初始化和工具发现测试；
- 鉴权错误和参数错误示例。

### 验收标准

- Claude Code 可以连接 `/mcp`；
- Claude Code 的 `/mcp` 页面显示服务正常；
- Claude Code 能发现 `moa_reason`；
- Claude Code 调用 mock 工具后能展示结果；
- 无 token 或错误 token 不能调用工具；
- Worker 不在错误响应中泄露内部堆栈。

## 阶段 2：Workers AI 单 Agent 适配

### 目标

先完成稳定的单模型调用，为后续 proposer 和 aggregator 提供统一接口。

### 工作项

- 选择首个已验证的 Workers AI 文本模型；
- 实现 `WorkersAIAdapter`；
- 统一模型调用参数和响应文本提取；
- 为每个模型维护输入格式、输出格式、上下文窗口、streaming 和 function calling 能力；
- 建立模型白名单；
- 对模型响应做 schema/contract test；
- 对上游异常分类：参数错误、模型不可用、限流、超时和内部错误；
- 实现有限重试，禁止无限重试；
- 记录单次模型调用耗时和状态。

### 交付物

- `src/workers-ai/adapter.ts`；
- `src/workers-ai/models.ts`；
- `src/workers-ai/normalize.ts`；
- 首个可运行模型配置；
- 单 Agent baseline 测试报告。

### 验收标准

- `moa_reason` 可以返回真实 Workers AI 模型结果；
- 模型响应可以稳定提取为文本；
- 上游错误可以映射到统一错误码；
- 客户端无法通过参数选择未加入白名单的模型；
- 单 Agent 请求具备请求 ID、耗时和错误日志。

## 阶段 3：两层 MoA 编排

### 目标

实现 MVP 核心链路：第一层并行 proposer，第二层 aggregator。

### 默认运行参数

```text
language = auto
mode = balanced
layer_count = 2
proposer_count = 3
include_trace = true
```

### 工作项

- 实现 profile 配置读取；
- 实现第一层 proposer prompt；
- 使用受控并发并行调用 proposer；
- 收集每个 proposer 的结果、状态、耗时和错误；
- 对成功结果进行长度限制和脱敏；
- 构建 aggregator 输入；
- 实现 aggregator prompt；
- 调用 aggregator 生成最终答案；
- 统一处理部分失败和全部失败；
- 实现总执行超时和调用次数预算。

### 聚合原则

Aggregator 必须：

- 批判性比较候选结果；
- 识别错误、冲突、幻觉和遗漏；
- 综合生成新答案，而不是简单选择或拼接候选；
- 保持用户要求的语言和格式；
- 不输出隐藏思维链；
- 对无法确认的内容标注不确定性。

### 交付物

- `src/moa/orchestrator.ts`；
- `src/moa/prompts.ts`；
- `src/moa/profiles.ts`；
- 两层 MoA 集成测试；
- 部分失败和全部失败测试。

### 验收标准

- 同层 proposer 确实并行执行；
- aggregator 可以收到上一层候选结果；
- 最终返回 `answer`；
- 一个 proposer 失败时，在成功数量达到阈值的情况下仍可继续；
- 全部 Agent 失败时返回 `ALL_AGENTS_FAILED`；
- 不会因单个模型失败无限重试。

## 阶段 4：中间 Agent 结果协议

### 目标

从 MVP 开始稳定返回中间 Agent 结果，避免后续修改 MCP 工具 schema。

### 返回结构

```json
{
  "request_id": "req_xxx",
  "answer": "最终 aggregator 答案",
  "intermediate_results": [
    {
      "layer": 1,
      "agents": [
        {
          "agent_id": "layer-1-agent-1",
          "role": "proposer",
          "model": "<model-id>",
          "status": "succeeded",
          "output": "候选答案",
          "duration_ms": 1200,
          "truncated": false
        }
      ],
      "aggregation_input": {
        "type": "candidate_summary",
        "content": "下一层收到的候选内容"
      },
      "aggregation_output": null
    },
    {
      "layer": 2,
      "agents": [
        {
          "agent_id": "layer-2-aggregator-1",
          "role": "aggregator",
          "model": "<aggregator-model>",
          "status": "succeeded",
          "output": "聚合答案",
          "duration_ms": 2800,
          "truncated": false
        }
      ],
      "aggregation_input": {
        "type": "candidate_summary",
        "content": "上一层候选内容"
      },
      "aggregation_output": "聚合答案"
    }
  ],
  "trace": {
    "total_duration_ms": 5000,
    "degraded": false
  }
}
```

### 工作项

- 实现 `intermediate_results` 构造器；
- 固定字段名和字段类型；
- 对每个 Agent 输出设置最大长度；
- 对总响应设置最大大小；
- 截断时设置 `truncated=true`；
- 保留失败 Agent 的状态和错误类别；
- 区分 `aggregation_input` 和 `aggregation_output`；
- 默认返回中间 Agent 文本；
- `include_trace` 只控制额外执行元数据，不关闭中间 Agent 结果；
- 脱敏 API token、内部路径、堆栈和敏感数据；
- 验证长响应仍是合法 MCP 工具结果。

### 验收标准

- Claude Code 收到最终答案和中间结果；
- 中间结果按 layer 顺序排列；
- 每个 Agent 都包含稳定的 ID、角色、模型和状态；
- 过长输出被明确截断，不会静默丢弃；
- 返回内容不包含隐藏思维链或鉴权信息。

## 阶段 5：配置、预算和区域

### 目标

实现服务端运行配置，使账户计划、预算、区域、模型和 profile 可配置，同时保留服务端硬限制。

### 配置项

```text
CLOUDFLARE_ACCOUNT_PLAN       # 默认 free
WORKERS_AI_BUDGET             # 单请求、每日或计费周期预算
WORKERS_AI_REGION              # 默认 US
MOA_DEFAULT_PROFILE            # 默认 balanced
MOA_MAX_LAYERS
MOA_MAX_PROPOSERS
MOA_MAX_INPUT_CHARS
MOA_MAX_OUTPUT_CHARS
MOA_MAX_TOTAL_RESPONSE_CHARS
MOA_REQUEST_TIMEOUT_MS
MOA_MAX_RETRIES
MOA_LOG_LEVEL
MOA_AUTH_TOKEN                # Secret
```

### 工作项

- 实现配置读取和类型校验；
- 支持 `free` 等账户计划配置；
- 支持预算配置和预算耗尽处理；
- 支持 `US` 等区域配置，默认美国；
- 实现 fast、balanced、quality profile；
- 根据账户计划和区域筛选可用模型；
- 客户端只能提交 `language`、`mode` 等允许的业务参数；
- 客户端不能覆盖模型 ID、硬限制和预算；
- 账户计划、区域或模型配置变更后，执行启动检查或部署前检查。

### 验收标准

- 不修改 Claude Code URL 和工具 schema 即可替换服务端模型；
- `free` + `US` 默认配置可以运行最小链路；
- 预算不足返回 `BUDGET_EXCEEDED` 或执行明确的受控降级；
- 参数不能绕过最大层数、最大 proposer 数量和总调用预算；
- 配置错误在 Worker 启动或首次请求时给出明确提示。

## 阶段 6：语言支持与安全加固

### 目标

完成中文、英文和中英混合任务支持，并满足上线前安全要求。

### 工作项

- 支持 `language=auto`、`zh-CN` 和 `en-US`；
- 未显式指定语言时识别任务主要语言；
- 在 proposer prompt 和 aggregator prompt 中注入语言约束；
- 测试中文、英文、中英混合和代码混合输入；
- 中间结果保留 Agent 原始语言；
- 最终答案按用户指定或识别出的主要语言输出；
- 完善输入大小、输出大小、并发、超时和重试限制；
- 检查鉴权绕过、模型白名单绕过和预算绕过；
- 验证模型输出不会被作为代码执行；
- 日志和错误响应脱敏；
- 明确日志及中间结果的清理策略。

### 验收标准

- 中文任务能返回中文最终答案；
- 英文任务能返回英文最终答案；
- 中英混合任务可以正常聚合；
- 不同 Agent 使用不同语言时，最终 aggregator 仍按主要语言输出；
- 安全测试没有发现 token、内部堆栈或敏感中间结果泄漏。

- 两层 MoA 正常返回；
- 中间 Agent 结果格式稳定；
- 中文、英文和中英混合测试通过；
- 鉴权、预算、限流、超时和响应大小测试通过；
- 生产模型在配置区域可用；
- 关键错误可通过 request ID 定位；
- 已准备上一版本回滚配置。

## 7. 配置与接口冻结计划

为了避免开发后期频繁修改 Claude Code 配置，以下内容应在阶段 1 结束时冻结：

- MCP URL 路径：`/mcp`；
- MCP Server 名称：`moa`；
- 工具名称：`moa_reason`；
- 工具输入字段：`task`、`language`、`mode`、`layer_count`、`proposer_count`、`include_trace`；
- 工具输出顶层字段：`request_id`、`answer`、`intermediate_results`、`trace`；
- 鉴权方式：Bearer Token；
- Claude Code 配置方式：CLI 和项目级 `.mcp.json`。

字段可以增加，但不应随意修改已有字段的含义。模型 ID、层数默认值和预算属于服务端配置，不应进入 Claude Code 固定配置。

## 10. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| MCP SDK 不兼容 Cloudflare Workers | 无法接入 Claude Code | 阶段 0 先做 runtime 验证；必要时使用轻量协议实现 |
| Workers AI 模型 schema 不一致 | 运行时错误 | 使用模型 adapter 和 contract test |
| 免费账户限额不足 | 无法完成多 proposer 并行 | 默认使用 fast profile，设置严格预算和调用上限 |
| 多层 MoA 延迟过高 | Claude Code 使用体验差 | 提供 fast/balanced/quality，优先减少层数 |
| 中间结果过大 | MCP 响应失败或上下文膨胀 | 单 Agent 和总响应截断，设置 `truncated=true` |
| proposer 全部失败 | 无最终结果 | 有限重试、受控降级并返回 request ID |
| 模型被下线或区域不可用 | 生产请求失败 | 模型白名单、部署前检查和备用模型配置 |
| token 泄漏 | 服务被滥用 | Cloudflare Secret、日志脱敏、token 轮换 |
| 提示词注入影响聚合 | 输出质量或安全下降 | 候选内容隔离、aggregator 明确批判规则、不执行模型输出 |
| 配置变更导致结果不一致 | 难以排障 | 记录配置版本、profile 和模型组合 |

## 11. 开发完成定义

只有同时满足以下条件，才认为 MVP 开发完成：

- Claude Code 可以稳定连接 `/mcp`；
- Claude Code 可以发现并调用 `moa_reason`；
- Worker 可以调用已验证的 Workers AI 模型；
- 两层 MoA 运行成功，同层 proposer 使用受控并行；
- 返回最终答案和各层中间 Agent 结果；
- 支持中文、英文和中英混合任务；
- 模型、账户计划、预算和区域可以由服务端配置；
- 鉴权、预算、并发、超时和输入输出限制生效；
- 部分失败和全部失败均有明确行为；
- 日志和返回结果不会泄露 token、内部堆栈或隐藏思维链；
- 单元测试、MCP 集成测试、Claude Code 端到端测试和基础离线评测完成；
- 已完成生产部署、烟囱测试和回滚验证。
