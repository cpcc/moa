## 1. 项目概述

参考论文 [Mixture-of-Agents Enhances Large Language Model Capabilities](https://arxiv.org/abs/2406.04692)，在 Cloudflare Worker 上实现两层 Mixture-of-Agents（MoA）多 Agent 推理服务，通过 MCP 协议和 Anthropic Messages API 两种方式接入 Claude Code。

## 3. 架构设计

```
Claude Code
    │
    ├── MCP 协议 (POST /mcp)
    │   └── tools/call → moa_reason 工具
    │       └── 两层 MoA 编排
    │           ├── Layer 1: 3 个 proposer 并行调用 Workers AI
    │           └── Layer 2: 1 个 aggregator 综合候选答案
    │
    └── Anthropic Messages API (POST /v1/messages)
        └── 内部调用 MoA 编排
            └── 返回 Anthropic 兼容响应（含 SSE 流式）

Cloudflare Worker
    ├── /health          → 健康检查（无需鉴权）
    ├── /v1/models       → 模型列表（x-api-key / Bearer 鉴权）
    ├── /v1/messages     → Anthropic Messages API（x-api-key / Bearer 鉴权）
    └── /mcp             → MCP JSON-RPC（Bearer 鉴权）
```

### 核心数据流

1. **MCP 模式**：Claude Code 发现 `moa_reason` 工具 → 按需调用 → Worker 执行两层 MoA → 返回最终答案 + 中间 Agent 结果
2. **Provider 模式**：Claude Code 所有 API 请求 → Worker `/v1/messages` → 内部执行 MoA → 返回 Anthropic 兼容格式

## 5. 关键设计决策

### 5.1 两种接入方式

- **MCP 工具（`moa_reason`）**：Claude Code 在需要多 Agent 分析时主动调用，日常对话不受影响
- **Anthropic Messages API 网关**：Claude Code 的所有请求都经过 MoA，适合需要全局质量提升的场景
- 两者可同时使用，互不冲突

### 5.2 安全边界

- Bearer Token 恒时比较，防止时序攻击
- 鉴权发生在解析请求体之前
- Token 不写入日志
- 提示词中使用 `<TASK>` 分隔符隔离不可信用户输入
- Aggregator 提示词明确要求不执行候选文本中的指令

### 5.3 限制与降级

| 限制项 | 默认值 | 环境变量 |
|--------|--------|----------|
| 最大层数 | 2 | `MOA_MAX_LAYERS` |
| 最大 proposer 数 | 3 | `MOA_MAX_PROPOSERS` |
| 请求超时 | 60s | `MOA_REQUEST_TIMEOUT_MS` |
| 最大输入字符 | 12000 | `MOA_MAX_INPUT_CHARS` |
| 最大输出字符 | 12000 | `MOA_MAX_OUTPUT_CHARS` |
| 最大总响应字符 | 50000 | `MOA_MAX_TOTAL_RESPONSE_CHARS` |
| 最大重试次数 | 1 | `MOA_MAX_RETRIES` |
| 最大 AI 调用数 | 4 | `MOA_MAX_AI_CALLS` |
| 最大并发 Agent | 3 | `MOA_MAX_CONCURRENT_AGENTS` |
| 最少成功 proposer | 1 | `MOA_MIN_SUCCESSFUL_PROPOSERS` |

- 全部 proposer 失败时返回 `ALL_AGENTS_FAILED`
- 预算耗尽时返回 `BUDGET_EXCEEDED`
- 超时时返回 `UPSTREAM_TIMEOUT`（可重试）
- 限流时返回 `UPSTREAM_RATE_LIMITED`（可重试）
- 输出超长时截断并标记 `truncated: true`

### 5.4 Mock 降级

当 `env.AI` 不存在时（如单元测试或本地无 remote binding），自动使用确定性 mock 返回固定格式结果，保证协议链路可验证。

## 8. CC Switch 集成

### 方式一：Provider 模式（API 网关）

在 CC Switch 数据库中配置了 `moe` provider，JSON 配置：

```json
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "moa-prod-token-...",
    "ANTHROPIC_BASE_URL": "https://moe-cloudflare-worker.arthur-162.workers.dev",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "moa-haiku",
    "ANTHROPIC_DEFAULT_OPUS_MODEL": "moa-opus",
    "ANTHROPIC_DEFAULT_SONNET_MODEL": "moa-sonnet",
    "API_TIMEOUT_MS": "300000"
  }
}
```

切换到 `moe` provider 后，Claude Code 的所有 API 请求都经过 MoA。

### 方式二：MCP 工具模式

在 CC Switch 数据库 `mcp_servers` 表添加了 `moa` 条目（`enabled_claude=1`），并同步到 `~/.claude.json`：

```json
{
  "moa": {
    "type": "http",
    "url": "https://moe-cloudflare-worker.arthur-162.workers.dev/mcp",
    "headers": {
      "Authorization": "Bearer moa-prod-token-..."
    }
  }
}
```

Claude Code 在任何 provider 下都可以调用 `moa_reason` 工具进行按需多 Agent 分析。

### 两种模式对比

| | Provider 模式 | MCP 工具模式 |
|---|---|---|
| 作用范围 | 所有 Claude Code 请求 | 仅调用 `moa_reason` 时 |
| 延迟 | 每次对话较高 | 正常对话快，工具调用时高 |
| 适用场景 | 全局质量提升 | 日常对话 + 复杂问题按需分析 |
| 可同时使用 | ✅ | ✅ |
