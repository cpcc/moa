# 基于 Cloudflare Workers AI 的 MoA 多 Agent 思考服务

> 文档版本：v0.1（需求初稿）  
> 更新时间：2026-07-16  
> 状态：待评审  
> 关联文件：[`moe.md`](./moe.md)

## 1. 文档目的

本文档定义一个部署在 Cloudflare Worker 上的多 Agent 推理服务。服务参考论文 **Mixture-of-Agents Enhances Large Language Model Capabilities**（arXiv:2406.04692）的分层 Mixture-of-Agents（MoA）方法，通过并行 proposer（提案 Agent）生成多个候选答案，再由 aggregator（聚合 Agent）逐层综合、校正并输出最终答案。

服务的核心目标是：

1. 使用 Cloudflare Workers AI 调用一个或多个可用模型。
2. 在 Worker 内完成多 Agent 的编排、并行调用、结果聚合与错误处理。
3. 对外提供面向 **Claude Code** 的 MCP 远程 HTTP 服务，使用户可以配置 Worker URL 和鉴权信息后，在 Claude Code 中使用该能力。
4. 保留模型、层数、并发度和策略的可配置性，为后续评测和成本优化提供基础。

## 2. 重要范围说明：Worker URL 的接入方式

### 2.1 目标接入方式

本项目默认将 Worker 实现为 **MCP Streamable HTTP Server**，而不是只暴露一个普通 JSON/REST 接口。

Claude Code 官方推荐通过远程 HTTP MCP Server 接入云端服务，配置示例：

```bash
claude mcp add --transport http moa https://<worker-domain>/mcp
```

也可以在项目 `.mcp.json` 中配置：

```json
{
  "mcpServers": {
    "moa": {
      "type": "http",
      "url": "https://<worker-domain>/mcp"
    }
  }
}
```

`streamable-http` 是 `http` 的别名。配置中不能只写 `url` 而省略 `type`，否则 Claude Code 会将其误判为 stdio 服务。

### 2.2 Claude Code 专属范围

本项目首版唯一面向用户的配置入口是 **Claude Code**。不提供独立 Web 配置页面，也不承诺兼容 claude.ai、Claude Desktop 或其他 MCP 客户端。用户只需要在 Claude Code 中配置 MCP Server；Worker 的模型、层数、并发和预算等运行配置由服务运营方在 Cloudflare 侧维护。

所有接入、验收和配置示例均以 Claude Code CLI 和项目级 `.mcp.json` 为准。

支持的主要配置方式：

```bash
claude mcp add --transport http moa https://<worker-domain>/mcp \
  --header "Authorization: Bearer <token>"
```

或在项目 `.mcp.json` 中配置：

```json
{
  "mcpServers": {
    "moa": {
      "type": "http",
      "url": "https://<worker-domain>/mcp",
      "headers": {
        "Authorization": "Bearer ${MOA_AUTH_TOKEN}"
      }
    }
  }
}
```

验收只要求 Claude Code 能完成：连接、MCP 初始化、工具发现、工具调用和错误展示。其他 MCP 客户端的差异不属于本期需求。

### 2.3 不在本期范围内的接入方式

以下接口可以作为后续扩展，但不能代替本期 MCP 接口：

- 仅接受 `POST /chat` 的普通 REST 接口；
- 仅兼容 OpenAI Chat Completions 的接口；
- 仅提供 Cloudflare Workers AI 的 `/ai/run` 或 `/ai/v1/responses` 代理；
- 仅提供 SSE 接口。

Claude Code 文档已将 SSE 作为旧式传输，优先使用 HTTP/Streamable HTTP。因此本项目应优先实现 `/mcp` MCP 端点；如 MCP SDK 或边缘运行时兼容性存在问题，可将 SSE 作为临时兼容层，但不能作为长期主协议。

### 2.4 Claude Code 与本 Worker 的职责边界

| 能力 | Claude Code | Cloudflare Worker |
|---|---|---|
| 理解用户当前意图 | 负责 | 接收已结构化的工具调用参数 |
| 决定是否调用 MoA 工具 | 负责 | 不负责用户侧工具选择 |
| MCP 协议通信 | 发起/接收 | 实现 Server 端 |
| 多 Agent 分层编排 | 不负责 | 负责 |
| 调用 Workers AI 模型 | 不直接负责 | 负责 |
| proposer 并行推理 | 不负责 | 负责 |
| aggregator 综合推理 | 不负责 | 负责 |
| 结果格式化和错误返回 | 消费工具结果 | 负责返回 MCP 工具结果 |
| 用户确认高风险外部操作 | 负责 | 本项目默认不执行外部副作用操作 |

本项目不是把 Claude Code 的内部思考过程暴露给 Worker，也不是通过 Worker 获取 Claude Code 的隐藏思维链。Worker 只接收 Claude Code 通过 MCP 工具传入的任务文本和配置，调用 Workers AI 生成候选与聚合结果，并向 Claude Code 返回最终答案及可审计的中间 Agent 结果。

## 3. 参考论文与设计提炼

论文的关键设计如下：

1. MoA 由多个层组成，每层包含多个 LLM Agent。
2. 第一层 Agent 针对同一个输入独立生成答案。
3. 下一层的 Agent 将上一层所有 Agent 的输出作为辅助信息，继续生成或综合答案。
4. Agent 可以在层内或跨层复用同一个模型。
5. Agent 可以承担 proposer 或 aggregator 角色：
   - proposer：提供有用、具有差异性的候选观点；
   - aggregator：批判性评估候选结果，并综合为更高质量答案。
6. 论文指出，模型多样性和 proposer 数量通常有利于质量提升，但更多层和更多模型会增加成本、延迟和首 token 等待时间。
7. 论文的聚合方式不是简单选择某一个候选答案，而是通过提示词让聚合模型综合多个候选答案。
8. 论文方法主要依赖提示词和推理接口，不要求微调模型。

### 3.1 本项目的工程化映射

| 论文概念 | 本项目实现 |
|---|---|
| 输入 prompt | MCP 工具 `moa_reason` 的 `task` |
| proposer Agent | Workers AI 上的并行模型调用 |
| aggregator Agent | Workers AI 上的聚合模型调用 |
| MoA layer | Worker 编排器中的一轮 proposer/aggregator |
| 多模型多样性 | 为不同 proposer 配置不同 Workers AI 模型或不同角色提示词 |
| Aggregate-and-Synthesize Prompt | Worker 内置的聚合系统提示词 |
| 最终输出 | 最后一层 aggregator 的文本结果 |
| 质量/成本权衡 | `layer_count`、`proposer_count`、模型配置和预算策略 |

论文中的评测数字仅作为研究背景，不作为本项目上线 SLA 或质量承诺。论文使用的模型、价格和服务环境与本项目不同，必须通过本项目自己的离线数据集进行重新评测。

## 4. 产品定位与用户故事

### 4.1 目标用户

- 已使用 Claude Code，希望通过一个远程工具增强复杂任务的分析质量的开发者；
- 需要多模型交叉分析、方案比较、代码/架构评审、研究摘要的个人或小团队；
- 希望将模型编排逻辑放在 Cloudflare 边缘侧，而不是在本地维护多模型调用脚本的用户。

### 4.2 用户故事

1. 作为 Claude Code 用户，我可以配置一个 Worker URL，让 Claude Code 发现 `moa_reason` 工具。
2. 作为用户，我可以让 Claude Code 对一个复杂问题调用 MoA，而不需要自己了解 Worker AI 的具体模型 API。
3. 作为用户，我可以指定输出语言、回答风格、质量模式或成本模式。
4. 作为用户，我可以看到本次执行的每一层 Agent 中间结果、使用的模型、调用数量以及是否发生降级。
5. 作为运维人员，我可以限制单次请求的层数、并发数、输入长度和总推理预算，避免异常成本。
6. 作为开发者，我可以替换某一层的模型或提示词，而不修改 Claude Code 端配置。

### 4.3 语言支持

首版必须支持中文和英文任务，并支持中英混合输入。语言可以由 Claude Code 通过 `language` 参数显式传入；未传入时，Worker 应根据任务内容进行语言识别，并默认使用任务的主要语言输出。

语言要求：

- `zh-CN`：中文输出；
- `en-US`：英文输出；
- `auto`：自动识别，默认值；
- 中文和英文混合任务：保持用户指定的主要语言，必要时保留代码、专有名词和原文引用；
- proposer 和 aggregator 的提示词必须包含语言约束，避免不同 Agent 使用不同语言导致聚合质量下降；
- 中间 Agent 结果应保留其原始语言，同时在最终聚合结果中统一输出语言。


### 5.1 MVP 必须包含

- 一个公开 HTTPS Worker；
- MCP Streamable HTTP 端点，建议路径为 `/mcp`；
- 中间 Agent 结果默认返回；
- 返回最终答案以及各层中间 Agent 结果；
- 中间结果包含 proposer 输出、aggregator 输入摘要和 aggregator 输出，且受长度与脱敏规则限制；
- 一个名为 `moa_reason` 的 MCP 工具；
- Workers AI binding；
- 至少两层 MoA：第一层并行 proposer，第二层 aggregator；
- 基于 `Promise.all` 的同层并行调用；
- 聚合提示词和候选答案隔离；
- 输入校验、超时、部分失败处理和统一错误结构；
- 基本鉴权；
- 基本日志、请求 ID、耗时和 token/调用统计（若模型响应提供）；
- 本地 Wrangler 开发、测试和部署说明；
- Claude Code 配置示例。

### 5.2 MVP 不包含

- 自动执行 shell、写文件、发邮件、修改数据库、转账等外部副作用操作；
- 将 Worker 作为 Claude 的完整替代模型端点；
- 读取 Claude Code 本地文件系统；
- 访问 Claude 的隐藏 chain-of-thought；
- 自动训练、微调或更新模型权重；
- 复杂长期记忆；
- 多租户计费系统；
- 论文级别的完整 benchmark 复现；
- MCP 外部工具扩展。

## 6. 功能需求

### FR-001：MCP 服务发现

Worker 必须实现 MCP 初始化和工具发现流程，使 Claude Code 能够通过 `/mcp` 连接并发现工具。

验收标准：

- `claude mcp add --transport http moa <WORKER_URL>/mcp` 可以添加服务；
- Claude Code 的 `/mcp` 页面可以看到服务已连接；
- Claude Code 可以发现 `moa_reason` 工具及其输入 schema；
- Claude Code 可以成功发起一次 `tools/call` 并展示最终答案；
- 未授权请求返回明确的 HTTP 鉴权错误，不泄露内部异常。

### FR-002：`moa_reason` 工具

工具用于请求一次 MoA 推理。建议输入 schema：

```json
{
  "type": "object",
  "additionalProperties": false,
  "required": ["task"],
  "properties": {
    "task": {
      "type": "string",
      "description": "需要多 Agent 分析并回答的任务"
    },
    "language": {
      "type": "string",
      "enum": ["auto", "zh-CN", "en-US"],
      "description": "输出语言；默认 auto，支持中文、英文和中英混合任务"
    },
    "mode": {
      "type": "string",
      "enum": ["quality", "balanced", "fast"]
    },
    "layer_count": {
      "type": "integer",
      "description": "MoA 层数；服务端仍需执行最大值限制"
    },
    "proposer_count": {
      "type": "integer",
      "description": "每层 proposer 数量；服务端仍需执行最大值限制"
    },
    "include_trace": {
      "type": "boolean",
      "description": "是否返回中间 Agent 结果和执行摘要；本项目默认返回，不返回隐藏思维链"
    }
  }
}
```

服务端默认值建议：

```text
language = auto
mode = balanced
layer_count = 2
proposer_count = 3
include_trace = true
```

服务端必须重新校验所有参数，不能信任 Claude Code 传入的范围。

### FR-003：MoA 编排

Worker 应按以下逻辑执行：

1. 生成请求 ID。
2. 规范化用户任务和运行配置。
3. 读取当前层模型配置。
4. 并行调用当前层的 proposer。
5. 收集成功结果和失败结果。
6. 对候选结果进行长度限制、脱敏和结构化封装。
7. 将候选结果注入下一层 aggregator/proposer 的聚合提示词。
8. 重复执行直到达到 `layer_count`。
9. 返回最终 aggregator 结果和执行摘要。

概念伪代码：

```ts
let context = task;
let trace = [];

for (const layer of layers) {
  const results = await Promise.all(
    layer.agents.map(agent => runWorkersAI(agent.model, buildPrompt(agent, context)))
  );

  const usable = results.filter(result => result.ok);
  if (usable.length === 0) throw new Error("all agents failed");

  trace.push(summarizeLayer(layer, results));
  context = buildAggregateInput(task, usable);
}

return { answer: context, trace };
```

实际实现中，最后一层应明确使用 aggregator 提示词，而不是简单把候选答案拼接后原样返回。

### FR-004：Workers AI 模型调用

Worker 必须通过 Workers AI binding 调用模型。Wrangler 配置示例：

```jsonc
{
  "ai": {
    "binding": "AI"
  }
}
```

TypeScript Worker 中通过 `env.AI` 访问 binding，并使用类似以下方式调用：

```ts
const result = await env.AI.run("@cf/openai/gpt-oss-120b", {
  input: prompt
});
```

具体模型 ID、输入字段、输出格式、上下文窗口、函数调用能力和价格必须以部署时的 Cloudflare 模型目录及对应模型页为准，不能在代码中假设所有模型具有相同 schema。

### FR-005：模型配置

模型配置支持由服务运营方调整。每个 profile 可以配置不同的 proposer、aggregator、层数和 Agent 数量。客户端只能选择服务端允许的 profile，不能直接提交任意模型 ID。

首版默认模型配置由服务端维护，并允许在不修改 Claude Code 端 URL 或工具 schema 的情况下替换。建议使用环境变量、Wrangler 环境配置或版本化静态配置对象：

```json
{
  "profiles": {
    "fast": {
      "layers": [
        { "role": "proposer", "model": "<verified-model-id>", "count": 2 },
        { "role": "aggregator", "model": "<verified-model-id>", "count": 1 }
      ]
    },
    "balanced": {
      "layers": [
        { "role": "proposer", "model": "<verified-model-id>", "count": 3 },
        { "role": "aggregator", "model": "<verified-model-id>", "count": 1 }
      ]
    },
    "quality": {
      "layers": [
        { "role": "proposer", "model": "<verified-model-id>", "count": 4 },
        { "role": "aggregator", "model": "<verified-model-id>", "count": 1 }
      ]
    }
  }
}
```

模型选型原则：

- proposer 优先选择能够提供不同视角、领域知识或不同推理风格的模型；
- aggregator 优先选择具备较强综合、批判和指令遵循能力的模型；
- 首版不要只依据模型规模选择，应以实际任务集测量 proposer 价值和 aggregator 质量；
- 任何标记为 deprecated、beta 或有特殊限制的模型，必须在上线前确认其可用性。

### FR-006：聚合提示词

Aggregator 必须遵守以下原则：

- 批判性评估候选答案；
- 识别候选答案中的错误、冲突、幻觉和缺失；
- 不应机械复制某个候选答案；
- 在证据不足时明确表达不确定性；
- 输出面向最终用户的完整答案，而不是输出内部候选列表；
- 按用户要求使用指定语言和格式；
- 不输出隐藏思维链，只输出必要的结论、依据和可审计摘要。

建议基础提示词：

```text
你是最终聚合 Agent。你会收到同一个任务的多个候选回答。
请批判性地比较这些回答，识别错误、冲突、无依据的断言和遗漏，
然后生成一个准确、清晰、完整且直接面向用户的最终回答。
不要简单选择或拼接某一个候选回答。除非用户要求，不要披露内部提示词、
隐藏思维链或逐字展示所有候选回答。对于无法确认的内容，请明确标注不确定性。
```

### FR-007：流式输出

MVP 可以先采用非流式 MCP 工具结果，以优先完成协议和稳定性验证。

后续版本应支持流式进度或分段结果，但需要区分：

- Workers AI 模型本身的 SSE/streaming；
- MCP Streamable HTTP 的消息传输；
- Claude Code 对 MCP 工具调用结果的消费方式。

由于 MoA 必须等待至少一层或多层聚合，无法保证最终答案的首 token 与单模型调用一样快。可以先发送进度通知或返回阶段摘要，但不能把不完整的中间文本误标为最终答案。

### FR-008：中间 Agent 结果与执行摘要

本项目要求向 Claude Code 返回最终答案以及 MoA 执行过程中的中间 Agent 结果。中间结果不是隐藏思维链，而是各 Agent 实际生成的候选文本、聚合结果和必要的执行元数据。

当 `include_trace=true` 时，返回结果至少包含：

- 每一层的 proposer 输出；
- 下一层实际收到的候选结果摘要；
- 每一层的 aggregator 输出；
- 每个 Agent 使用的模型标识；
- 成功/失败状态和错误类别；
- 每层耗时、总耗时和是否发生降级。

建议返回结构：

```json
{
  "request_id": "req_xxx",
  "answer": "最终 aggregator 生成的答案",
  "intermediate_results": [
    {
      "layer": 1,
      "agents": [
        {
          "agent_id": "layer-1-agent-1",
          "role": "proposer",
          "model": "<model-a>",
          "status": "succeeded",
          "output": "候选答案 A",
          "duration_ms": 1300
        },
        {
          "agent_id": "layer-1-agent-2",
          "role": "proposer",
          "model": "<model-b>",
          "status": "succeeded",
          "output": "候选答案 B",
          "duration_ms": 1500
        }
      ],
      "aggregation_input": {
        "type": "candidate_summary",
        "content": "传递给下一层的候选结果或脱敏摘要"
      },
      "aggregation_output": null
    },
    {
      "layer": 2,
      "agents": [
        {
          "agent_id": "layer-2-aggregator-1",
          "role": "aggregator",
          "model": "<aggregator-model>",
          "status": "succeeded",
          "output": "聚合后的答案",
          "duration_ms": 3100
        }
      ],
      "aggregation_input": {
        "type": "candidate_summary",
        "content": "上一层候选结果"
      },
      "aggregation_output": "聚合后的答案"
    }
  ],
  "trace": {
    "total_duration_ms": 7600,
    "degraded": false
  }
}
```

中间结果要求：

- `include_trace` 的默认值改为 `true`；
- 默认返回完整中间 Agent 文本，但必须执行单个结果和总响应大小限制；
- 若超过响应大小限制，应保留每个 Agent 的截断输出，并标记 `truncated=true`，不能静默丢弃；
- 必须明确区分 `output`、`aggregation_input` 和 `aggregation_output`；
- 中间结果应保持稳定的层级、Agent ID、角色、模型和状态字段，便于 Claude Code 展示与后续调试；
- 中间结果中的用户敏感数据需要脱敏或按配置处理；
- 中间结果不得包含 API token、内部 stack trace、可绕过鉴权的信息或模型隐藏思维链；
- Claude Code 端主要展示 `answer`，并可在用户要求或工具结果较长时展示 `intermediate_results`。

### FR-009：部分失败和降级

当同层部分 proposer 失败时：

- 只要成功结果数量达到最小阈值，允许进入下一层；
- 记录失败模型、错误类别和耗时，但不把内部堆栈返回给用户；
- 当全部 proposer 失败时，可根据 mode 执行一次受控降级：降低层数、切换已验证模型或直接返回可解释错误；
- 不得无限重试；每个请求必须有最大重试次数和总超时。

### FR-010：鉴权

Worker 必须要求调用方携带鉴权信息。MVP 可以采用静态 Bearer Token，后续再接入 OAuth 或更细粒度的用户身份体系。

示例：

```bash
claude mcp add --transport http moa https://<worker-domain>/mcp \
  --header "Authorization: Bearer <token>"
```

要求：

- token 使用 Cloudflare Secret 管理，不写入源码和 Wrangler 配置仓库；
- 日志中不得输出完整 token；
- 未授权、token 无效、token 过期必须分别返回明确状态；
- 必须考虑 token 轮换；
- 如服务公开部署，必须配合请求频率限制和滥用防护。

### FR-011：输入输出限制

服务端必须配置并执行：

- 最大任务字符数/字节数；
- 每个候选结果的最大字符数；
- 聚合上下文最大大小；
- 最大层数；
- 每层最大 proposer 数量；
- 单请求总执行时间；
- 单请求最大 Workers AI 调用次数；
- 单请求最大重试次数。

超限请求应在调用模型前快速失败，并返回可操作的错误说明。

## 7. 非功能需求

### NFR-001：可靠性

- 同层 proposer 必须并行执行，避免不必要的串行等待；
- 单个模型失败不能导致整个请求无条件失败；
- 关键错误需要带 `request_id`；
- 对 Cloudflare/Workers AI 的临时错误采用有限指数退避；
- 对参数错误、鉴权错误、模型不存在等不可重试错误不得重试。

### NFR-002：性能

目标值（待压测确认）：

- MCP 初始化成功率 ≥ 99%；
- 无模型调用时的 Worker 协议响应 P95 < 500ms；
- balanced 模式端到端 P95 在可接受范围内，具体数值由压测确定；
- 同层并发度不能超过 Cloudflare 和 Workers AI 的实际限制；
- 必须记录每层耗时、模型耗时和总耗时。

论文已明确指出，多层聚合会提高 TTFT；因此本项目应提供 fast/balanced/quality 模式，而不能只追求最高质量。

### NFR-003：安全

- 所有外部请求必须使用 HTTPS；
- 仅允许明确的模型白名单，禁止客户端任意指定模型 ID；
- 通过 schema 校验工具参数；
- 防止超大输入、恶意嵌套、提示词注入导致的资源滥用；
- 不执行来自模型输出的 shell 命令或任意代码；
- 对日志、追踪和错误响应执行脱敏；
- 不将用户任务和候选答案无期限持久化；
- 对跨租户配置和缓存进行隔离；
- 如未来增加外部工具，必须为副作用操作增加显式确认和权限边界。

### NFR-004：可观测性

每次请求至少记录：

- request ID；
- 时间戳；
- 模式；
- 层数、每层调用数；
- 使用的模型白名单 ID；
- 成功/失败数量；
- 每层和总耗时；
- 是否降级；
- Workers AI 返回的可用 usage 信息；
- 错误类别。

日志不得记录完整敏感任务、Bearer Token 或模型隐藏思维链。默认只保留截断后的哈希或摘要。

### NFR-005：可维护性

- MCP 协议层、MoA 编排层、Workers AI 适配层、鉴权层和日志层分离；
- 模型输入/输出转换集中在 adapter 中；
- 提示词版本化；
- 每个 profile 有独立配置和测试；
- 模型下线或替换不应要求修改 Claude Code 端 URL。

## 8. 建议技术架构

```text
Claude Code
    │
    │ MCP Streamable HTTP /mcp
    ▼
Cloudflare Worker
    ├── Authentication middleware
    ├── MCP initialize / tools/list / tools/call
    ├── Request validation and rate limiting
    ├── MoA Orchestrator
    │     ├── Layer 1: proposer A ─┐
    │     ├── Layer 1: proposer B ─┼─ parallel Workers AI calls
    │     ├── Layer 1: proposer C ─┘
    │     ├── Layer 2: aggregator/proposer
    │     └── Final aggregator
    ├── Usage / trace summarizer
    └── Structured MCP result
          │
          ▼
    Claude Code presents the answer to the user

Workers AI binding: env.AI
```

### 8.1 组件职责

1. **MCP Transport**：处理 HTTP 请求、MCP 消息、协议错误和连接生命周期。
2. **Auth Middleware**：校验 Bearer Token、租户和权限。
3. **Request Validator**：校验工具输入、默认值和资源预算。
4. **MoA Orchestrator**：根据 profile 生成执行计划并控制层级。
5. **Workers AI Adapter**：统一不同模型的输入和输出格式。
6. **Prompt Builder**：生成 proposer、aggregator、降级提示词。
7. **Result Normalizer**：提取文本、处理不同模型响应格式。
8. **Trace/Usage Collector**：生成可观测信息和脱敏执行摘要。

### 8.2 中间结果传递与展示

Worker 必须将中间 Agent 结果作为 `moa_reason` 的结构化返回内容传回 Claude Code。Claude Code 负责在工具调用结果中接收这些内容，并根据用户任务和返回大小进行展示。

传递要求：

- `answer` 是最终答案，优先供 Claude Code 继续处理或展示；
- `intermediate_results` 保存按层组织的 Agent 输出；
- 每个 Agent 结果包含 `agent_id`、`role`、`model`、`status`、`output` 和 `duration_ms`；
- 每层记录 `aggregation_input` 与 `aggregation_output`；
- 中间结果默认启用，`include_trace` 只控制额外执行元数据是否返回；
- 若内容过长，Worker 必须截断并设置 `truncated=true`；
- Claude Code 配置不负责模型和 MoA 参数，Worker 统一负责执行限制和返回格式。

## 9. Cloudflare 实现注意事项

### 9.1 Binding 与部署

- Cloudflare 账户计划、Workers AI 预算和目标区域均支持配置；
- 当前默认配置为免费账户、可调整预算、美国（US）推理区域；
- 部署前需要根据实际账户计划确认 Workers AI 的可用模型、请求限制、并发限制和计费规则；
- 模型目录会变化，部署脚本和运行时都不应依赖已废弃模型。

### 9.2 模型 API 差异

Cloudflare 当前模型目录中的模型能力并不完全一致。部分模型支持 function calling、reasoning、vision、batch 或 streaming，部分模型不支持。因此：

- 将模型能力登记为配置元数据；
- 运行前只选择已验证的模型；
- 不能把某个模型的 `messages` schema 直接套用到所有模型；
- 应为模型响应编写 adapter 和 contract test；
- 上线前验证同步、流式和上下文窗口行为。

### 9.3 并发与资源

论文结构天然适合同层并行，但 Cloudflare Worker 和 Workers AI 具有实际的请求、执行时间、并发、上下文和配额限制。MVP 必须以压测数据确定安全并发，而不是根据 proposer 数量无限扩张。

建议：

- `Promise.all` 只用于受控数量的同层调用；
- 使用 semaphore 或批次并发控制；
- 每一层设置独立超时；
- 为单请求设置总预算；
- 记录“理论并行调用数”和“实际成功调用数”；
- 当预算不足时优先减少层数，再减少最终 aggregator 质量，具体策略通过 profile 控制。

### 9.4 流式与 TTFT

模型支持流式并不意味着 MoA 可以立即流出最终答案。只要下一层依赖上一层完整输出，Worker 就必须等待上一层完成。因此 MVP 采用完整结果聚合更简单；后续可以研究：

- chunk-wise aggregation；
- 仅在第一层完成后发送阶段进度；
- aggregator 增量读取 proposer 输出；
- 对质量模式和快速模式设置不同层数。

## 10. MCP 接口需求

### 10.1 必须支持的 MCP 方法

MVP 至少支持：

- `initialize`；
- `notifications/initialized`；
- `tools/list`；
- `tools/call`。

如采用 MCP SDK，需确认其在 Cloudflare Workers runtime 下不依赖 Node.js 专有 API、长驻进程、TCP socket 或本地文件系统。

### 10.2 `tools/list`

返回 `moa_reason` 及稳定的 JSON Schema。Schema 变更需要版本管理，避免 Claude Code 端出现不兼容调用。

### 10.3 `tools/call`

`moa_reason` 调用结果必须包含可供 Claude Code 展示的最终答案和中间 Agent 结果。推荐将 `answer` 作为主要文本，将 `intermediate_results` 作为结构化附加内容；当响应较长时，Claude Code 可根据用户请求展示全部或截断后的中间结果。

当请求失败时，应使用 MCP 的错误结果表达，并在文本中包含：

- 可读错误原因；
- request ID；
- 是否建议重试；
- 不暴露内部 stack trace、密钥或未脱敏上游响应。

## 11. 配置需求

### 11.1 环境变量/Secret

建议至少包含：

```text
MOA_AUTH_TOKEN                 # 使用 Cloudflare Secret 管理
MOA_DEFAULT_PROFILE            # fast / balanced / quality
MOA_MAX_LAYERS
MOA_MAX_PROPOSERS
MOA_REQUEST_TIMEOUT_MS
MOA_MAX_INPUT_CHARS
MOA_LOG_LEVEL
```

模型 ID 和复杂配置可放在版本控制的静态配置中，但 token 等敏感值必须使用 Secret。

### 11.3 Cloudflare 运行配置

Cloudflare 账户计划、Workers AI 预算和推理区域均属于可配置项，不得在业务代码中硬编码。配置应由服务运营方维护，并通过环境变量、Wrangler 环境配置或 Secret/配置中心注入。

推荐配置项：

```text
CLOUDFLARE_ACCOUNT_PLAN       # free / paid 等，具体枚举以实际部署环境为准
WORKERS_AI_BUDGET             # 单请求、每日或计费周期预算
WORKERS_AI_REGION              # 默认 US，可配置其他已支持区域
```

默认配置：

```text
CLOUDFLARE_ACCOUNT_PLAN = free
WORKERS_AI_REGION = US
WORKERS_AI_BUDGET = configurable
```

运行时必须根据当前账户计划和区域选择已验证模型，并在预算耗尽时拒绝请求或执行受控降级。账户计划和区域变化后，需要重新执行模型可用性、限额、延迟和成本验证。


### 11.2 配置优先级

建议优先级：

```text
服务端硬限制 > 租户限制 > profile 默认值 > 工具调用参数
```

客户端不能通过工具参数绕过服务端硬限制。

## 12. 错误码建议

| 错误码 | 含义 | 是否建议重试 |
|---|---|---|
| `UNAUTHORIZED` | 缺少或无效鉴权 | 否，先修复配置 |
| `FORBIDDEN` | 无权使用该 profile/能力 | 否 |
| `INVALID_ARGUMENT` | 工具参数不合法 | 否 |
| `INPUT_TOO_LARGE` | 输入超限 | 否，缩短输入 |
| `BUDGET_EXCEEDED` | 超过调用预算 | 否，降低模式或规模 |
| `MODEL_UNAVAILABLE` | 模型不存在、下线或暂不可用 | 视情况 |
| `UPSTREAM_TIMEOUT` | Workers AI 超时 | 有限重试 |
| `UPSTREAM_RATE_LIMITED` | 上游限流 | 按 retry-after/退避策略 |
| `PARTIAL_FAILURE` | 部分 Agent 失败但仍生成结果 | 通常不需要 |
| `ALL_AGENTS_FAILED` | 全部 Agent 失败 | 有限重试 |
| `INTERNAL_ERROR` | 未分类内部错误 | 有限重试并记录 request ID |

## 13. 质量评测需求

### 13.1 离线评测集

首版应准备至少三类任务：

1. 技术问答与架构设计；
2. 长文本总结与事实一致性；
3. 代码解释、代码审查或方案比较。

每类任务应包含：

- 用户输入；
- 参考答案或评分标准；
- 单模型 baseline；
- 2 层 MoA；
- 3 层 MoA；
- 不同 proposer 数量；
- fast/balanced/quality 配置。

### 13.2 评测指标

- 事实正确性；
- 指令遵循；
- 完整性；
- 可读性；
- 幻觉率；
- 代码问题发现率（如适用）；
- 平均/ P95 延迟；
- 首 token 延迟；
- Workers AI 调用次数；
- 输入/输出 token；
- 单请求成本；
- 部分失败率；
- 最终答案与 proposer 的重复率。

MoA 只有在质量提升足以抵消额外成本和延迟时才应在某类任务中默认启用。否则应保留单模型或 fast profile。

## 14. 验收标准

### 14.1 接入验收

- 使用一条 `claude mcp add --transport http` 命令完成配置；
- Claude Code 能发现并调用 `moa_reason`；
- 未授权访问无法调用工具；
- MCP 初始化、工具发现和工具调用均符合协议。

### 14.2 功能验收

- 2 层 MoA 能成功返回最终答案；
- 同层 proposer 确实并行执行；
- aggregator 能看到上一层候选结果；
- 任一 proposer 失败时，仍能按策略降级；
- 全部 Agent 失败时返回可读错误和 request ID；
- `include_trace` 可控制是否额外返回执行元数据；中间 Agent 结果本身默认返回；

### 14.3 安全验收

- token 不出现在日志、错误和返回结果中；
- 客户端不能指定未授权模型；
- 输入和预算限制有效；
- 模型输出不会被当作 shell 或代码执行；
- 日志经过敏感信息脱敏。

### 14.4 性能验收

- 在目标并发下没有未控制的请求爆炸；
- 超时和限流能够被捕获；
- 能区分模型耗时、层耗时和总耗时；
- fast profile 明显低于 quality profile 的平均调用次数或延迟。

## 15. 实施阶段

### Phase 0：协议验证

- 选择 Cloudflare Workers MCP SDK 或兼容实现；
- 在 Worker 中实现 `/mcp`；
- 暴露一个固定返回值的 `moa_reason`；
- 使用 Claude Code 完成连接和工具调用。

### Phase 1：单层并行 proposer

- 接入 Workers AI binding；
- 配置一个已验证文本模型；
- 并行调用 2–3 个 proposer；
- 返回候选合并结果和基础统计。

### Phase 2：第二层 aggregator

- 增加聚合提示词；
- 实现候选答案长度限制和失败处理；
- 返回最终答案；
- 加入 balanced profile。

### Phase 3：配置化与安全加固

- 增加 fast/quality profile；
- 接入 Secret 鉴权、限流和硬预算；
- 增加模型 adapter、contract test 和日志脱敏。

### Phase 4：评测与优化

- 建立离线数据集；
- 对层数、宽度、模型多样性进行对比；
- 评估成本/质量/延迟 Pareto 前沿；
- 决定默认 profile。

### Phase 5：可选增强

- 流式阶段进度；
- chunk-wise aggregation；
- 更完整的 MCP 资源或 prompts；
- 多租户、OAuth、持久化评测和配额系统。

## 15.1 详细开发计划

### 开发原则

- 先验证 Claude Code 到 MCP Worker 的完整链路，再实现多 Agent 推理；
- 先实现固定配置和稳定返回，再实现模型、预算、区域和 profile 的动态配置；
- 中间 Agent 结果从第一版开始纳入返回协议，避免后续修改 MCP 工具 schema；
- 不实现 MCP 外部工具，不执行 shell、文件写入、数据库修改或其他外部副作用操作；
- 每个阶段都必须有可运行的验收结果，不能只完成代码而没有 Claude Code 端验证。

### 阶段一：项目初始化与协议选型

**目标：** 建立可部署的 Cloudflare Worker，并确定适用于 Workers runtime 的 MCP Streamable HTTP 实现。

**开发任务：**

1. 初始化 TypeScript Worker 项目和 Wrangler 配置；
2. 配置 `env.AI` Workers AI binding；
3. 确认 MCP SDK 在 Cloudflare Workers 中不依赖 Node.js 专有 API、TCP socket 或本地文件系统；
4. 设计 `/mcp` 路由和统一错误响应；
5. 定义 `Env` 类型和配置读取接口；
6. 建立基础 lint、格式化和单元测试命令。

**交付物：**

- 可通过 `wrangler dev` 启动的 Worker；
- `/mcp` 路由骨架；
- Wrangler 环境配置模板；
- MCP SDK 选型记录；
- 项目 README 初稿。

**完成标准：**

- Worker 可以本地启动；
- MCP 协议初始化请求可以收到合法响应；
- 没有调用 Workers AI 时，协议测试可以稳定通过。

### 阶段二：Claude Code 接入与固定工具

**目标：** 让 Claude Code 可以连接 Worker、发现并调用 `moa_reason`。

**开发任务：**

1. 实现 `initialize`、`notifications/initialized`、`tools/list` 和 `tools/call`；
2. 定义稳定的 `moa_reason` 输入 schema；
3. 初期使用固定 mock 结果验证完整调用链；
4. 增加 Bearer Token 鉴权中间件；
5. 编写 Claude Code CLI 配置命令和项目级 `.mcp.json` 示例；
6. 验证 Claude Code `/mcp` 页面中的连接状态和错误展示。

**交付物：**

- 可被 Claude Code 发现的 `moa_reason` 工具；
- Bearer Token 鉴权；
- `.mcp.json` 配置示例；
- MCP 协议集成测试；
- 未授权、参数错误和内部错误的返回规范。

**完成标准：**

```bash
claude mcp add --transport http moa https://<worker-domain>/mcp \
  --header "Authorization: Bearer <token>"
```

执行后，Claude Code 能够连接、发现 `moa_reason` 并展示 mock 工具结果。

### 阶段三：Workers AI 单 Agent 适配层

**目标：** 在不引入 MoA 复杂性的情况下，先完成可靠的单模型调用。

**开发任务：**

1. 选择并验证首个 Workers AI 文本模型；
2. 实现 `WorkersAIAdapter`；
3. 统一模型输入、输出文本提取和错误分类；
4. 增加模型白名单；
5. 增加模型能力配置，记录上下文窗口、输入格式、输出格式、streaming 和 function calling 能力；
6. 为模型适配器编写 contract test；
7. 实现请求超时、有限重试和上游错误映射。

**交付物：**

- 单 Agent Workers AI 调用链；
- 模型白名单和模型配置文件；
- `WorkersAIAdapter` 和响应 normalizer；
- 上游错误处理；
- 单模型 baseline 测试结果。

**完成标准：**

- Claude Code 通过 `moa_reason` 能得到真实模型答案；
- 模型不存在、超时、限流和无效请求能被区分；
- 不同模型响应格式不会泄漏到 MCP 协议层。

### 阶段四：MoA 两层编排

**目标：** 实现 MVP 的核心 MoA 链路：第一层并行 proposer，第二层 aggregator。

**开发任务：**

1. 实现 profile 配置读取；
2. 实现第一层 proposer 并行调用；
3. 实现 proposer 结果收集、状态记录和长度限制；
4. 实现 aggregator prompt builder；
5. 将第一层候选结果传给第二层 aggregator；
6. 实现 aggregator 输出规范化；
7. 处理部分 proposer 失败和全部失败；
8. 生成最终 `answer`。

**建议初始配置：**

```text
language = auto
mode = balanced
layer_count = 2
proposer_count = 3
include_trace = true
```

**交付物：**

- MoA Orchestrator；
- proposer prompt builder；
- aggregator prompt builder；
- 两层运行时 trace；
- 最终答案和中间结果结构。

**完成标准：**

- 同层 proposer 确实并行执行；
- aggregator 可以接收到上一层候选结果；
- MCP 返回 `answer` 和 `intermediate_results`；
- 一个 proposer 失败时仍能在阈值允许的情况下生成结果；
- 全部 Agent 失败时返回 `ALL_AGENTS_FAILED`。

### 阶段五：中间结果协议与 Claude Code 展示

**目标：** 固化中间 Agent 结果格式，并确保较长结果可安全传回 Claude Code。

**开发任务：**

1. 实现按层组织的 `intermediate_results`；
2. 记录 `agent_id`、`role`、`model`、`status`、`output` 和 `duration_ms`；
3. 记录 `aggregation_input` 和 `aggregation_output`；
4. 实现单 Agent 输出和总响应大小限制；
5. 截断时设置 `truncated=true`；
6. 对 token、stack trace、内部路径和敏感数据做脱敏；
7. 验证 Claude Code 能处理最终答案和中间结果；
8. 验证结果过长时不会破坏 MCP 响应格式。

**完成标准：**

- `include_trace=true` 时返回完整的中间 Agent 结果和执行元数据；
- 中间结果超长时被显式截断；
- Claude Code 可以继续使用最终答案，不会将中间结果误认为新的工具调用；
- 不返回隐藏思维链或认证信息。

### 阶段六：配置、预算和美国区域

**目标：** 让账户计划、预算、区域、模型和 MoA profile 都可配置，同时保持服务端硬限制。

**开发任务：**

1. 实现 `CLOUDFLARE_ACCOUNT_PLAN` 配置；
2. 实现 `WORKERS_AI_BUDGET` 配置；
3. 实现 `WORKERS_AI_REGION` 配置，默认 `US`；
4. 实现 fast、balanced、quality profile；
5. 实现单请求调用预算和周期预算检查；
6. 预算不足时返回 `BUDGET_EXCEEDED` 或执行受控降级；
7. 根据账户计划和区域筛选模型白名单；
8. 记录配置版本，便于定位不同部署行为。

**完成标准：**

- 不修改 Claude Code URL 和工具 schema 即可替换服务端模型配置；
- 客户端不能通过参数绕过服务端模型和预算限制；
- 免费账户、默认美国区域下至少有一个 profile 可运行；
- 账户计划或区域变化后可以重新执行配置验证。

### 阶段七：中英文能力与安全加固

**目标：** 达成中文、英文和中英混合任务要求，并完成上线前安全检查。

**开发任务：**

1. 实现 `language=auto`、`zh-CN` 和 `en-US`；
2. 在 proposer 和 aggregator prompt 中注入语言约束；
3. 测试中文、英文、中英混合和代码混合任务；
4. 确认中间结果保留原始语言，最终答案统一主要语言；
5. 完善输入长度、响应大小、并发和超时限制；
6. 完善 token、敏感数据和错误堆栈脱敏；
7. 验证模型输出不会被执行为 shell 或任意代码；
8. 明确日志和中间结果的暂存与清理策略。

**完成标准：**

- 中文和英文任务均能得到可读的最终结果；
- 中英混合任务不会因语言切换导致聚合失败；
- 安全测试未发现鉴权绕过、预算绕过和敏感信息泄漏。

### 开发过程中的暂停条件

遇到以下情况时不得直接进入生产发布：

- MCP SDK 无法在 Cloudflare Workers runtime 稳定运行；
- 当前账户计划、区域或模型白名单无法完成最小可运行链路；
- 中间 Agent 结果导致响应超过平台或 Claude Code 可接受大小；
- 预算限制无法可靠阻止过量调用；
- 无法确认日志和中间结果中的敏感数据处理方式；
- Claude Code 无法稳定完成 MCP 初始化、工具发现或工具调用。

## 18. 关键实现结论

1. **Claude Code 连接的不是任意 Worker URL，而是实现 MCP 协议的 Worker URL。**
2. **Worker 内部才负责调用 Cloudflare Workers AI，并执行 MoA 的多层、多 proposer、aggregator 编排。**
3. **Cloudflare Workers AI 的模型 API 和能力存在差异，必须通过模型白名单和 adapter 隔离。**
4. **MoA 的核心收益来自多样化候选和聚合，不是简单重复调用同一个模型并拼接文本。**
5. **更多层和更多 proposer 会增加质量潜力，也会增加成本、延迟和 TTFT，因此必须提供预算和模式控制。**
6. **MVP 应先完成 MCP 接入和 2 层非流式链路，再扩展流式、动态路由和复杂工具能力。**
